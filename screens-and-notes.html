Scope: This is the mobile view of the existing StikkyData v2 webpage. Most of what you see is v2 data rendered for a phone (RENDER). The only genuinely new work is the mobile shell/layout and the photo/note capture flow (BUILD). Asset-first. Photos are a single placeholder cropped per slot — swap real field photos in later.

1Map — the front door render + shell

opens here
  • This map is a TERRITORY display — its job is "look how much we own." It opens framed on the customer's whole multi-state footprint. Top bar = StikkyData brand + the America First customer chip + search + the map/list toggle (same bar on every screen).
  • Pins PILE UP where the customer is dense (home turf) and thin out elsewhere — the pile of pins IS the message. Each ATM is its own pin; they overlap on purpose. Do NOT cluster them and do NOT space them evenly — clustering hides the density, even spacing erases the scale. A 1,000-ATM customer must look enormous next to a 50-ATM one.
  • Pin = the StikkyData app's pushpin: a green ball with a glossy highlight on a slate stick.
  • Peek sheet at the bottom — "N ATMs · M need attention." Drag/tap it up and it becomes the list over the map. Search filters it.
  • Tap a pin → a card: swipe its photos in place, then Show me (open asset) or Drive there (maps).
  • Data = v2's real asset coordinates. For big fleets (PNC ~8k) the pins stream in progressively (see footer) — all of them, never collapsed into a count.

2List render + shell

top-bar map/list toggle
  • The same fleet as photo-forward cards — big photo on top, status flag (orange "N open" / green "Clean"), ATM id, location, "Serviced N ago."
  • Reached via the map/list toggle in the top bar. Search filters by id or location; "Recently viewed" first.
  • Tap a card → the asset. Data = v2 fleet list. Big fleets → infinite-scroll + virtualize.

3Asset detail render + shell

tap a pin / card
  • Header: back chevron · ATM id + address · three icon actions — Directions · Camera · Share (Camera is the only one that opens the camera).
  • Hero photo with count "1/7", date, slot name, tap-to-enlarge. Below it the slot strip (Main, Receipt, Card Reader, Monitor, Keypad, Fascia, Fee Notice) — tap to swap the hero.
  • Status row: last service + open-issue count.
  • Tabs — Issues / History / Info. Issues = action items with the v2 verbs Email / Comment / Remove. History = visit timeline. Info = asset details. All v2 data.

  • Swipe between slots; a single tap toggles the chrome away so it's just the photo. No boxed buttons, no side arrows.

9Triage — clear a group new build · its own door

"N to review" chip → teams (accordion) → a team or type
  • A separate mode, NOT part of map/asset. Browse stays calm and asset-first; issue lists never bleed onto the map or asset screens. Reached from the "N to review" chip in the top Zillow-style controls (no bottom nav). Opens as a full-screen takeover with a Done (✕) back to the map.
  • Purpose: clear the standing flagged-issue backlog in ~10–15 min/month. Most flags are noise (e.g. ~35 monitor flags, ~2 real) — today that triage is a phone call; this replaces it with thumb-swipes.
  • Landing = an accordion of teams (mirrors the StikkyData dashboard). First look is just the three teams collapsed — Diebold Nixdorf · America First CU · Facilities — each with its open count. No photos on this interim list; the photo lives on the triage cards, not here.
  • Expand a team → launch the whole team or a single type. Expanding shows Triage all issues (N) plus the issue types with counts (Diebold: Monitor 45 · Receipt 10 · ATM Lighting 3; America First: ADA 145 · ATM Condition 33 · Braille 12 · ATM Out of Service 4; Facilities: Drive-Up Lanes 15). They clear one group at a time — you do NOT triage everything together. (Key correction from the first pass.)
  • A group = a card stack of just that type (each card the same component photo), or, for "Triage all", the team's types back-to-back (each card shows its own type + photo). A progress bar runs along the top.
  • Two modes, one stack (borrowed-wins read, 2026-06-22): the bin you open IS the mode — a noisy bin (Monitor 45) is a fast "take out the trash" run; a small bin is careful review. Swipe the card ← to Excuse, → to Escalate (buttons also work) — the swipe is what makes the trash run fast. An optional comment rides into the escalation email.
  • Three actions: Excuse (dismiss as not-real — v2's "excuse" action), Escalate (it's real → that team), and "Can't tell — skip for now" (Skip is back per Michael; the skipped item stays open for next time). Auto-advances; ~2s "Excused · Undo" safety. Continuous flow — finishing one type rolls straight into the next; the header shows the current type, and a single type offers "Next: <type>".
  • Multi-photo: an issue can carry several photos — the card shows a count badge; tap the photo or the ATM id to open the full set in a swipe viewer.
  • Undo: a brief "Excused · Undo" (~2s) after each action — fat-finger safety on a live backlog.
  • Escalate = one email at the end of the group. Escalations (with their comments) batch within the group; the summary shows "Email N to Diebold Nixdorf" with the team's default address beneath it. Tapping opens a pre-filled draft — the To is the dashboard's per-department routing address but is editable, so the user can redirect to a manager, a specific vendor rep, or anyone before sending (Michael's pick, 2026-06-22). Nothing auto-sends. Later option: have v2's existing routing send it automatically (config already exists).
  • Summary + safety net: the group summary tallies excused · escalated · left open; tap "review" on the excused count to see everything you just dismissed and Reopen any that shouldn't have been (the batch-safety move from photo-culling apps). Excused/escalated clear from the open count; skipped stay open.
  • Dispositions EXISTING issues only — never creates new action items (that's deferred web-side work).

  • Icon buttons 38×38 everywhere.
  • No bottom nav — Zillow model (map is home, the peek sheet is the list).

Key decisions what was chosen and why — so the build doesn't re-open settled questions

Map = a territory display, not a tidy map
ATMs pile up where a customer is dense; that footprint across the states IS the value. Real multi-state spread, no even spacing.
No clustering
Every pin shows individually — clustering into count bubbles hides the very density that's the point.
Pin = the app's round pushpin
Green ball + slate stick + a soft shadow. Matches the live StikkyData app, not a generic teardrop.
Drawing on photos — cut
Not worth the build time (Claire, 2026-06-22).
Ticketing — pulled for now
v2 ticketing isn't done; removed the ticket flow and the Tickets tab. Revisit when v2 ships it.
Triage grouped by team → issue type
Mirrors the StikkyData dashboard; you clear one group at a time, not a mixed pile. Accordion landing, no photos on the list.
Swipe + Skip + "review excused"
Borrowed from email / photo-culling / moderation queues: the bin you open IS the mode (fast trash vs careful review). Swipe for speed, Skip for "can't tell," review-excused as the safety net.
Escalation = editable pre-filled email
Default recipient = the team's dashboard routing address, but editable so it can go to a boss / rep / anyone. Nothing auto-sends. Later: let v2's routing send it automatically.
Transitions: view changes cross-fade (~0.2s); the peek sheet drags & snaps collapsed/expanded; bottom sheets slide up over a dimmed overlay; the photo viewer toggles chrome on tap. (Motion only shows on a real device.)

Cut for now (Claire, 2026-06-22): draw-on-photo (not worth the build time) and ticketing (v2 ticketing isn't done) — both removed; no Tickets/Reports tab on mobile.
Still to build / placeholders: real ATM photos (swap into the per-slot arrays); large-fleet handling for big customers (background-stream the map pins, infinite-scroll + virtualize the list — no clustering); confirm v2's "excuse" action + vendor emails for Triage (§8); wire Escalate to v2 route-to-default later if/when it exists (email is the interim).
Out of scope on mobile: Reports, fleet-wide dashboards/filters, any StikkySurveys project/board patterns, system-sent email (Send goes from the user's own phone).

Mobile — Fidelity Notes

The deliberate choices that make this feel finished. The Feature Spec Bible covers what each screen does; this covers how it should look and behave. Every value below is intentional — not a default to "clean up." Each entry lists the exact value, why it matters, and where it lives in the code.

Three handoff rules (read first)

  1. The CSS file is the spec. Lift stikkydata-mobile.css verbatim as the first epic — don't re-implement it from ticket prose. The polish lives in the stylesheet; carry the stylesheet.
  2. Polish is the foundation, not the finish line. Build the design system + shared styles first, so every screen inherits them. Left to the end, they're the first thing cut.
  3. Parity gate on every UI ticket. Definition of Done = "open the live prototype and the build side-by-side at 390px — they match." Sign off against the hosted URL.
Brand & color 3 colors, no more
Node-graph mark + wordmark lockup (the frog is retired)Don't drop
The mark is a small network of connected asset-nodes, paired with the wordmark, in the app bar on every screen. The old zen-frog reads playful/startup and undercuts a bank buyer — it's gone. Swap in the official mark SVG when available; the lockup slot is ready. Frog also still needs removing on the v2 login (outside this prototype).
.lockup / .bmark — every .appbar
One green — three color roles totalDon't drop
Green = brand and success/done. Orange = action / active / counts. Slate = text & neutrals. Two near-identical greens read as an accident — they were unified to one.
Green · brand + done · #1a7a1c Orange · action · #FF9800 / #E65100 Slate · text · #0f172a / #64748b / #94a3b8
:root — --brand-green, --accent, --pass (= brand green)
Icons: one set, one stroke weight
stroke-width: 2 on every functional icon
Mixed stroke weights are the tell of an unfinished UI. The brand mark is the only intentional exception at 1.6. Keep line-style, sentence case, no emoji.
all inline SVG icons
Design tokens build to the grid, don't hardcode
Spacing grid, radii, type ramp are variables — use themDon't drop
--sp-1…8 = 4·8·12·16·20·24·32
--r-sm/md/lg = 8·11·14 (sharp; a pill is deliberate)
--t-h1/h2/body/sub/cap = 20·16·14·12.5·11
Hierarchy comes from size + weight, not color or decoration. Two weights: 400 body, 700/800 emphasis. Reach for a token, not an arbitrary px — that's what keeps the rhythm consistent across screens.
:root — --sp-*, --r-*, --t-*
Photos — the "pop" never edit the file
Photos pop at render time, not in the pixelsDon't drop
filter: saturate(1.22) contrast(1.1) brightness(1.02)
This is the Zillow/Airbnb trick — the browser renders the photo richer; the file is never altered (critical for "the photo is the proof"). Remove it and every photo goes flat. Applied to all photos.
.hero img, .atm-img, .zcard-photo img, .pcard .pc-photo img
Depth scrim on the hero photoDon't drop
linear-gradient(to bottom, rgba(0,0,0,.20), transparent 17%, transparent 56%, rgba(0,0,0,.48))
A faint darkening at top + bottom edges so the photo reads three-dimensional, not pasted-on — and it keeps the white chips legible over a bright sky. It sits under the chips (chips are z-index 2).
.hero::after
Photos are real field shots — keep them realIntent
Don't auto-enhance, beautify, or stock-swap. The authenticity is the product. The only treatment is the render-filter + scrim above, plus a consistent crop/aspect. Swap the prototype's stand-in photos for the real per-slot field photos.
slot_photos/ → inlined by the build
Hover affordance guarded so touch never sticks
Tappable things lift / pop on hover; photos zoomDon't drop
cards: translateY(-2px) + box-shadow 0 6px 18px rgba(15,23,42,.14)
buttons: translateY(-1px) · photos: scale(1.05)
Signals "this does something" on any pointer device. All of it is wrapped in @media (hover: hover) — without that guard, a phone gets stuck hover states after a tap. The guard is load-bearing, not optional.
@media (hover: hover) { … } block
Expanded photo (lightbox) the iOS-rotation minefield
On a real phone the photo goes true full-screen and rotatesDon't drop
@media (hover:none) and (pointer:coarse), (max-width:480px), (max-height:480px)
.lb { position: fixed; inset: 0; height: 100dvh }
Target the device, not the width. A max-width:480px guard silently breaks in landscape — a rotated phone is ~844px wide, so it stops matching and the photo snaps back into the portrait frame. (hover:none) and (pointer:coarse) holds in both orientations. On desktop it stays in-frame.
.lb media query
100dvh, not 100vhDon't drop
100vh overshoots behind the iOS Safari toolbar and jumps on scroll. 100dvh (dynamic viewport) is the fix — with a 100vh fallback line first for old iOS.
.lb (inside the media query)
Safe-area insets on the chromeDon't drop
top: max(52px, calc(env(safe-area-inset-top) + 14px)) · left/right via env(safe-area-inset-*)
Keeps the close / share / count clear of the notch & Dynamic Island in both orientations. Requires viewport-fit=cover on the meta tag (already set).
.lb-fab, .lb-head, .lb-slot
Background scroll-locked while open
body.lb-open { overflow: hidden }
Stops the page behind the photo from rubber-banding / scrolling through on iOS. Toggled in openLB()/closeLB().
body.lb-open
ATM number + address live in the photo headerIntent
So a screenshot of just the photo is self-identifying (people screenshot to share). Header = ATM-#### + full address; the slot name + photo count sit bottom-left. Same reasoning as the sticky survey header.
.lb-head (#lb-atm / #lb-addr), syncLB()
Load-bearing gotchas subtle, but they break things
The search field must be allowed to shrinkDon't drop
.tb-search { min-width: 0 }
Without it, the search flex item won't shrink and shoves the map/list toggle off the right edge of a 390px screen — the button looks "broken" because it's literally unreachable. This is a one-line fix for a real, easy-to-reintroduce bug.
.tb-search
Views cross-fade with opacity, not display:none
Inactive views keep their size, so the Leaflet map doesn't go grey when you return to it. Still call map.invalidateSize() on switch back to the map (already wired in showView).
.view / .view.active, showView()
Branded loading splash
A pulsing green node-graph mark covers the empty-map flash, then removes via display:none after load (an opacity-only fade was overridden by a transition quirk — the hard remove is intentional).
.map-splash, hideSplash()

    /* swipe the card: left = Excuse, right = Escalate; a small drag that doesn't cross the threshold

    function wireSwipe(card) {
      if (!card) return;
      let sx = 0, sy = 0, dragging = false, moved = false, downT = null;
      card.addEventListener('pointerdown', e => { sx = e.clientX; sy = e.clientY; dragging = true; moved = false; downT = e.target; card.style.transition = 'none'; try { card.setPointerCapture(e.pointerId); } catch (_) {} });
      card.addEventListener('pointermove', e => {
        if (!dragging) return;
        const dx = e.clientX - sx, dy = e.clientY - sy;
        if (Math.abs(dx) > 6 || Math.abs(dy) > 6) moved = true;
        if (Math.abs(dx) > Math.abs(dy)) { card.style.transform = `translateX(${dx}px) rotate(${dx * 0.03}deg)`; card.classList.toggle('to-excuse', dx < -36); card.classList.toggle('to-escalate', dx > 36); }
      });
      card.addEventListener('pointerup', e => {
        if (!dragging) return; dragging = false;
        const dx = e.clientX - sx, dy = e.clientY - sy;
        card.style.transition = ''; card.classList.remove('to-excuse', 'to-escalate');
        if (Math.abs(dx) > 92 && Math.abs(dx) > Math.abs(dy)) {
          const act = dx > 0 ? 'escalate' : 'excuse';
          card.style.transform = `translateX(${dx > 0 ? 520 : -520}px) rotate(${dx > 0 ? 18 : -18}deg)`; card.style.opacity = '0';
          setTimeout(() => triageDo(act), 140);
        } else {
          card.style.transform = '';
          if (!moved && downT && downT.closest && (downT.closest('.tg-photo') || downT.closest('.tg-atm'))) openTgPhotos();
        }
      });
      card.addEventListener('pointercancel', () => { dragging = false; card.style.transition = ''; card.style.transform = ''; card.classList.remove('to-excuse', 'to-escalate'); });
    }
